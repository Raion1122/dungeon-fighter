## 1. 結論（何が起きて、何が起きないか）

- 幾何は **`buildMap()` 末尾のマスク1ブロック**で作る。`MAP_H` / `ROOMS` / `CORRIDORS` の定数は1バイトも変えない。
  歩行可能行を **row 13-15 の3行だけ**に絞り、それ以外を tile=2 にマスクする。
- カメラは **縦を完全固定**する。`camTargetY` の代入は L3643 の1箇所のみなので、直後の無条件上書きで
  隊列bbox / CENTER_BIAS / cameraTightFocus をすべて無効化できる。「空が見えるか」が確率ではなく**恒等式**になる。
- **絵を作る前に、可視性を機械的に証明する。** アセット生成は STEP4 まで一切行わない。
- 味方の被ダメージは **0 のまま**。範囲魔法の救済は候補窓の拡張だけで行う。

**実現しないこと**は §7 に列挙。特に「横持ちの空は最大22%で Golden Axe の約40%には届かない」点は着手前の合意事項。

## 3. 過去の失敗を覆す根拠

前回（`1849dd6` / `f8a89ec`）は空+遠景を実装したのち、**iPhone横持ちで 0/67列・最大0px** という実測で全撤去した。
[index.html:4316-4325] の恒久メモが真因を記録している。失敗要因は**独立に3つ**ある。

| # | 要因 | 本計画での解決 |
|---|---|---|
| 1 | カメラが北端へ行かない（隊列クランプ L3625-3636 が主人公を追う） | `camTargetY` を usableH のみの関数で**無条件上書き**。分散 0 を assert できる |

| 3 | **未露出の第3要因**: row 0-12 は tile===2 なので Pass1a [L4447-4458] が `SPR_CEILING` を敷き詰める | 空は「描かれない」のではなく**暗い天井で塗り潰される**。`FIELD_MODE && ty < HORIZON_ROW` で continue |

> 前回は最大0pxだったため、**第3要因はまだ一度も露出していない**。壁だけ直しても暗い帯が出て終わる。

**第4の再発リスク（y軸版）**: `CLOUDS` の y は 920-1720 に集約されている [L4337付近]。

### STEP 0 — 受動カメラレコーダ（本体ロジック変更ゼロ / アセット0）★ここから着手

**目的**: 「実プレイのカメラ」を初めて観測可能にする。前回の全損は `__probe.setCam` による**強制カメラ**でしか
測っていなかったことに起因する。`natCam` は lerp [L3662] も shake [L3686] も飛ばしており、
**実際に描画に使われたカメラ値をこれまで一度も観測していない。**

**変更**
- `renderWorld()` の冒頭（= 最終的な camX/camY で実際に描く唯一の合流点）に検証用シームを1本:
  ```js
  if (window.__camTrace) window.__camTrace.push({ camX, camY,
      focus: !!camFocus, tight: cameraTightFocus, phase: currentPhase });
  ```
  ⚠ `renderWorldWithShake` [L3683-3689] は shake が無いと早期に `renderWorld()` を呼ぶので、
  **シームは renderWorld 側に置く**（L3686 直後に置くと無シェイク時のフレームを丸ごと取りこぼす）。
  `window.__camTrace` 未定義時は完全な no-op（既定 undefined）。
- `tools/driver_field_step0.js` を新設。**`freeze` も `quiesce` もかけない**
  （rAF も `setInterval(moveEnemies, 30)` [L11705] も生かしたまま実プレイを走らせる）。
  `generatedScenario`（familyId=caravan-escort）を `evaluateOnNewDocument` で注入し、3ウェーブ完走まで放置して trace を回収。

**測るもの（すべて数値で持ち帰る）**
1. `skyPx = HORIZON_Y - camY` を各構成で計算した**全サンプル最小値**（平均・中央値は禁止）
2. **母集団健全性**: `phase==="combat"` ≥1件 / `focus:true` ≥1件 / `tight:true` ≥1件 / shake発生 ≥1件。
   0件なら測定自体が無効
3. **遮蔽コスト**: 歩行帯 {2,3,5}行 × HUD {据置, 奪還} で、交戦中の生存ユニットと馬車が画面外/HUD下に入る割合
4. **★馬車の占有タイル数**（§7-6 の未解決リスク。3行帯を何行塞ぐか）
5. **帯の切断率**: 毎手番、帯の各列について全行が `getUnitOccupiedTiles` [L11474-11518] に含まれるかを走査し、
   切断列が1つでも存在するフレーム比率
6. `bottomHud` の再計算箇所 L3579 / L3721-3723 / L3735 / L6921 が同じ値を返すか（単一ソース化漏れ検出）

**撤退スイッチ**: `window.__camTrace` を定義しなければ完全 no-op。**見積り: S**

### STEP 6 — 移動デッドロック救済（RNG非依存・単独出荷可）

**「aStar への斜め移動追加」は却下。** 実コード検証の結果:
- 攻撃可否は隣接タイルでなく `tileChebyshev` [L13141] + `melee:{tiles:1}` [L13131] で判定
  → **斜めに立っているだけで melee 射程内。接敵枠は元から8近傍。**「包囲が4方向に制限される」は誤り
- L13279 `let goalTX = pTX, goalTY = pTY;` が初期値で、4近傍が全滅してもゴールは対象タイル自身にフォールバックし、
  aStar は [L11419-11421] でゴールを通行可扱いにする → `CAN'T ADVANCE` には落ちず「対象へ押し込む」挙動になる
- 斜めを足しても**列が全行埋まっていれば通れない**。斜めは迂回歩数の短縮でしかなく、切断そのものには効かない

**採用する修正**: `pathIgnoreAllies` フォールバック [L12556-12582]（現在は探索中の heroAI 専用）と同型の第2候補
`aStar(..., null)` を、戦闘移動3箇所 — `playerAdvanceOneTile` [L13228-13235] /
`allyAdvanceTowardPoint` [L18183-18189] / `enemyAdvanceOneTile` [L13292-13297] — に入れ、
それも null のときだけ `CAN'T ADVANCE` を出す。

**検証の主指標は `CAN'T ADVANCE` ではない**（ゴールフォールバックに吸収されて過少計上される）。
主指標 = **切断イベント率**（STEP0 の測定項目5と同じ定義）。
実験: 帯幅{3,5} × 救済{なし,あり} で PT4+敵6・200手番×20試行。
判定: **3行+救済ありで切断率 <1% なら行数を増やさず済む（＝空を96px余分に取れる）。** **見積り: M**

## 5. 範囲魔法の修正（決定: 味方誤射なし・窓拡張のみ）

### 前提の訂正（実コード読解済み）

**① 味方には元々一切当たらない。** ダメージループが `enemiesInArea(...)` しか回らない。全6箇所で確認:

```
18868  const affectedIdxs = enemiesInArea(best.tx, best.ty);        // スリープ
19018  const affectedIdxs = enemiesInArea(best.tx, best.ty, 3, 3);  // ファイアボール
19331  const affectedIdxs = enemiesInArea(best.tx, best.ty, 5, 5);  // アイスストーム
20882  const affectedIdxs = enemiesInArea(best.tx, best.ty, 3, 3);  // ヘイル・オブ・ソーンズ
20975  const affectedIdxs = enemiesInArea(best.tx, best.ty, 5, 5);  // 矢の斉射
```

`partyInArea` [L18947-18962] は**着弾点を選ぶときの門番としてしか使われていない**。
→ 慶彦さんの「着弾点さえあれば打てる仕様に」がそのまま最適解であり、バランスへの影響はゼロ。

**② 着弾点に walkable 条件は元々無い**（壁上でも空中でも撃てる）。真因は `partyInArea` の**2値拒否権**。

**③「9呪文」は同型ではない。** 箱型 origin 探索を持つのは**5つだけ**:

| 型 | 呪文 | 救済可否 |
|---|---|---|
| 箱（origin可変） | fireball L19002 / iceStorm L19315 / hailOfThorns L20867 / conjureVolley L20960 / L21244 | **窓拡張が効く** |
| 直線 | lightningBolt L19096-19118（術者起点 **8方向**。「4方向」は誤り） | ❌ origin自由度ゼロ |
| 円錐 | burningHands L19195-19224 / coneOfCold L19419（術者起点4方向、`coneTiles` は step3 で横幅5） | ❌ origin自由度ゼロ |
| splash判定 | L21162（`if(!partyInArea(pTX-1,pTY-1,3,3))` — 探索ですらない） | ❌ |

**④「100%不発」は過大。** 5x5 で味方が西/北にいる配置では現行コードのままクリーン候補が出る。降格率は配置依存。

### 採用アルゴリズム（窓拡張のみ）

```js
function pickAoeOrigin(caster, tTX, tTY, S, rangeTiles) {
  const span = S - 1;
  let best = null, bestFoes = 0;

  // ★現行は dx,dy ∈ [-(S-1), 0] なので「箱が必ずターゲットを含む」。
  //   3行帯では前衛がターゲットに隣接するため全候補が味方入りになる。
  //   窓を [-(S-1), +(S-1)] へ拡張し「ターゲットを含まない箱」を許可する
  //   = 前衛が敵前列を抑えている間に、術者がその東側の敵後列へ箱を置ける。
  for (let dy = -span; dy <= span; dy++)
  for (let dx = -span; dx <= span; dx++) {
    const c = { tx: tTX + dx, ty: tTY + dy };
    const cx = c.tx + (S - 1) / 2, cy = c.ty + (S - 1) / 2;   // 箱中心

    // ★窓拡張とセットで必須。現行は射程/LoSをターゲットにしか見ていないので、
    //   窓を広げただけだと射程外へ飛ぶ。
    if (tileChebyshev(caster.tx, caster.ty, cx, cy) > rangeTiles) continue;
    if (!hasLineOfSight(caster.tx, caster.ty, cx, cy)) continue;

    if (partyInArea(c.tx, c.ty, S, S)) continue;              // ← 拒否権は維持（誤射なし）
    const foes = enemiesInArea(c.tx, c.ty, S, S).length;
    if (foes > bestFoes) { best = c; bestFoes = foes; }
  }
  return best;   // null なら従来どおり allyBasicAttack へ降格
}
```

- **安全な着弾点がある限り、挙動は現行と同一**（上位互換）
- 直線・円錐・splash の3系統は**救済されない**。降格が残ることを承知で出す
- **すべて `IS_FIELD_THEME` ゲート**（`FIELD_MODE` ではない。`?field=0` は描画だけを戻す契約）。
  既存6シナリオは現行の `dx,dy∈[-(S-1),0]` のまま＝挙動もRNG消費順も1バイト不変

### 敵AoE の非対称について（重要な訂正）

- **`FIELD_ENEMY_AOE_SIZE_CAP` は完全な no-op**。L22553 `const blast = 3, half = 1;` で既に3固定、
  `hydraFireBreath` も `def.breathBlast || 3` かつデータ側 **L5187 / L5500 / L5634 の全3箇所が `breathBlast:3`**。
  5x5の敵AoEは1つも存在しない → 削除
- **「役割反転版ヘルパーで対称化」も成立しない**。ダメージループ [L22579-22611] は `allies` と `player` しか舐めず、
  **敵が敵を巻き込む経路がコード上に存在しない**。反転側の拒否権が恒真になり、探索が
  「PC最大命中の箱選択」に退化して**敵AoEが現状より強化される**（悪化）
- 真の対称化には敵→敵ダメージ経路の**新設**が必要。行わないなら「対称化した」と呼ばないこと

## 7. この計画が実現しないこと（着手前の合意事項）

1. **真の縦圧縮（遠いものを縦に潰す擬似3D）は実装しない。** 空を「上に足す」だけで奥行きを表現する。
   2026-07-18 会議の却下理由②「奥行きが読めない」は**未解決のまま残る**
2. **ユニットの深度ソートは行わない。** HPバーは body 直下の兄弟で z1-7 は満杯。
   3行帯では縦重なりが現状より増え、**奥の敵のHPバーが手前のキャラの上に浮く**。実機で最初に目に付く可能性が高い
3. **歩行可能集合は実質 12-18行 → 3行に変わる。**「ROOMS 定数を触らない」「caravan-road 専用」は実装手法の話であって、
   ゲーム体験が変わらないという意味ではない。2026-07-18 で却下された帯幾何の親戚であることを承知の上で進める
4. **横持ちの空は 86px(22%)** で、Golden Axe 実機（224px中約90px = 約40%）には届かない。
   到達するには歩行帯2行（＝切断率の悪化）が必要
5. **WORLD_SCALE（ズーム）は実装しない。** キャラを 96→54px に縮めるのは
   「大きいキャラが浅い地面帯を歩く」という Golden Axe の絵の本体と逆行する。
   ただし退路として、s 導入箇所（renderMap 可視範囲6箇所 / `screen→world` 逆変換は **L11254-11255 の1箇所のみ** /
   `getBoundingClientRect`・`elementFromPoint` の当たり判定は皆無 / DOMラッパー）をコメントで残す
6. **⚠ 護衛対象の馬車が帯を塞ぐ問題は未解決。** 馬車は `isObjective` の敵ユニット [L5771-5785] として `enemies[]` に
   居座りタイルを占有する。3行帯の中央に停まれば残り2行、`displaySize` 次第では実質1行になり街道が分断される。
   **護衛クエスト専用の改修でありながら護衛対象が最大の障害物になり得る** — STEP0 で占有タイル数を必ず測る
7. **「面白いか」は測れない。** STEP0-1 が証明するのは「空が画面に入っている」ことだけ。
   3行帯のオートバトルが見て面白いかは STEP3 のスクショと慶彦さんの判断でしか決まらない
8. **`--no-verify` は使わない。** index.html を触る全コミットで
   `py tools/add_changelog.py "<b>見出し</b> — 説明"` を実行する（pre-commit フックが強制）
9. **別ファイル新設はしない。** `resizeCanvas()` [L2922] が `fxCompact()` [L4057] を関数ホイストで参照しており、
   分割するとロード時 ReferenceError で即死する
