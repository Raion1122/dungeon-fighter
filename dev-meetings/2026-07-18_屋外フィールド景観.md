## ⚠️ 仕様書 v1.0 の扱い（重要）

`field_visual_spec.md` は**横スクロール型ベルトアクションの 2.5D 背景**を前提に書かれており、実装は**見下ろしグリッドマップ**である。以下は**そのままでは実装されない**:

| 仕様書の記述 | 判定 | 理由 |
|---|---|---|
| 800×480 固定画面 / 地平線 y=248 / 立ち位置ライン y=336 | ❌ 成立しない | 実装は可変ビューポート。`computeUiMetrics` L3514 の compact 判定が `h <= 480` と等号を含むため、800×480 にすると必ず ui-compact に落ちる |
| LAYER 4「石柱を左右に2本ずつ」「崖で画面を挟む」 | ❌ 成立しない | 横から見た構図。見下ろしでは左右に柱を置けない |
| §4 パース演出ルール（手前100%→奥65%） | ⏸ 保留 | 深度スケールは STEP4（任意）でのみ関係 |
| §7 スタイル指定 `16-bit SNES/arcade, vibrant colors, clean pixel art, high contrast` | ❌ **破棄** | 既存スプライトは実物確認で「暗く彩度の低い厚塗りペインタリー半リアル」。この指定で生成すると背景だけ鮮やかになりキャラが浮く。**25枚作り直しの事故になる** |
| §3 の5フィールド（遺跡/草原/山岳/砂漠/深森）のテーマ・配色パレット | ✅ **使える** | 土地テーマと色設計はそのまま活きる |
| §4 ライティング統一ルール（光源方向・メインカラー・影色） | ✅ **使える** | 画面ティントとして実装可能 |
| §5 アニメ仕様（雲の横スクロール、草の揺れ、蜃気楼） | 🔄 翻訳して使える | 雲＝遠景レイヤーの横流し、として実装 |
| §2 LAYER 1/2/3（空・雲・遠景パララックス） | 🔄 **翻訳して使える** | 見下ろしでは「マップ外周の天井 void タイル」を空/遠景に差し替える形で実現する ← **本計画の中核** |

**結論**: 仕様書は「土地テーマ・配色・光の設計図」として使い、「画面レイアウトの設計図」としては使わない。

### ⚠️ 描画で踏んではいけない地雷
- **`renderLighting` の early-return で素の `return` は禁止**（L3837-3842）。clear しないと最後の暗幕フレームが残り、「画面を回転させたら直る」再現性の低いバグになる
- **暗幕は canvas と DOM の2レイヤーある**。`renderLighting` だけ止めると「明るい地面に敵だけ透明」「見えているのに戦闘が始まらない」になる。`isEnemyVisibleToParty()` L11233 も `__daylight` を見る
- `resizeCanvas()` は `computeUiMetrics()` の**後にもう一度**呼ぶ必要がある（L3540-3543）。順序を崩すとスマホで `lightScale=0.5` が効かなくなり、**機能バグにならないので気付けない**
- resize/orientationchange リスナは**1本だけ**（L2908-2909）
- **`resizeCanvas()` @L2922 の同期トップレベル呼び出しが `fxCompact()` @L4037 を参照している**（関数ホイスト依存）。ファイル分割は不可能ではないがロード順が壊れやすい

### 戦闘エンジン（今回は触らない。ただし影響ゼロの根拠として記録）
- 判定コア（`playerSingleAttack` L15654 / `damageEnemy` L10345 / `defeatEnemy` L10523）は**座標を一切使わない**
- 位置の正典は**ピクセル**。タイル座標は `Math.floor(px/TILE_SIZE)` でその場導出（保持するのは罠/宝箱/檻のみ）
- 距離メトリックは `tileChebyshev()` L12981（呼び出し47箇所）、エンカウントは `Math.hypot` < `engagePx`
- **幾何を変えないので、これらは1バイトも触らない**

### 今回は回避されるが、幾何を変えたら踏む地雷（記録用）
- **5×5 タイルの AoE 呪文が実在する**（コーン・オブ・コールド L19162、オーク呪術師 L20807）。帯を狭くすると `partyInArea(...,5,5)` が常に true を返し、**当該呪文が永久に撃たれなくなる**。エラーもログも出ず「魔法使いが仕事してない」としてだけ現れる
- **z-index 1〜7 は満杯**: `.groundFx`=1 / `.fxFireAura`=2 / `.actionFlash`=3 / `.trap`=3 / `.roomCage`=3 / `.roomChest`=4 / `.fxFirePillar`=4 / `.beast-roaring`=5(!important) / `.enemyHpBar`=6 / `.fxCastOrb`=6 / `.allyHpBar`=7。しかも **HPバーはスプライトの子ではなく body 直下の兄弟**なので、深度ソートすると奥のキャラのHPバーが手前に浮く
- `spawnMinionNearBoss` L22039 は**8近傍1リングしか探さない**。狭い帯だと召喚が空振りし、ボスが静かに弱体化する
- **`enemies[]` から要素を削除禁止**（L22921）。`enemyElements[]` が添字並列。削除は `alive=false` で表現
- **`units` 配列をイテレート中に push しない**（L14234, L14248-14249 に「唯一の正しい場所」と明記）
- **RNG ストリームの順序が決定論契約**（L17780-17782, L17216）。条件判定より先に乱数を引くとヘッドレス検証の全アサートが壊れる

### iOS Safari の現実（ノエル実測）
- **横持ち 844×390 は compact 判定に落ち、HUD が 173px = 画面の 44%。ゲーム描画に残るのは約 217px**
- 縦持ち 390×844 は縦660px使えるが、**横390px = 道が4タイル分しか見えない**
- `renderLighting` のコストの89%は放射グラデのラスタ画素数。**`__daylight` で丸ごと消えるので、屋外モードは描画負荷がむしろ下がる**
- **本作は今すでに 33〜60Hz で世界を再描画し続けている**（`moveEnemies` の setInterval 30ms L11705、L24346 で無条件起動 → `updatePositions()` → L9606-9626 で同期描画）。「常設ループは無い」は FALSE だった
- `assets/` は既に **142MB**（PNGだけで127MB）。`caravan_road_floor.png` = 2.79MB、`caravan_road_wall.png` = 2.82MB。**新規PNGは1枚500KB以下（pngquant 通し）を受け入れ基準にする**

### アセットの事実
- ✅ **既存スプライトは横視点**。`warrior_walk.png` も `goblin_anim.png` もサイドビューの歩行サイクル（ゴブリンは歩行3段＋突き攻撃1段の多段シート）。約80体分が流用可能
- ⚠️ 既存の部屋1枚絵（`room_goblin-mine_1.png` 等）は **painterly（厚塗り・写実寄り）**。トンマナ基準はこれ
- ⚠️ **`scene_*.png` はパララックス背景に流用不可**。実物確認で**一点透視の正面構図**（道が奥へ収束し左右の木が額縁を作る）。側面カメラとは真逆で横タイルも視差もできない。「アセットが6枚もう有る」という前提で計画を立ててはいけない

# 📋 開発計画書：屋外フィールド景観 STEP1〜

## 方針（背骨）

> **マップ幾何・戦闘エンジン・タイルデータ構造を1バイトも変えない。**
> 変えるのは `renderMap()` の描画パスと、屋外テーマ時の分岐だけ。

これにより:
- 既存6シナリオへの影響が**構造的にゼロ**（分岐条件が屋外テーマのみ）
- 5×5 AoE 死・z-index 再設計・団子化・iPhone横持ち問題を**すべて回避**
- 駄目なら差分を revert するだけで戻る

## STEP1（最初の実装 / アセットゼロ）

**ゴール**: 屋外シナリオを開いたとき「天井のある地下室」に見えなくなる。背景PNGは1枚も作らない（全部 canvas の手続き描画）。

### 実装内容

1. **`FIELD_MODE` の導入**
   - `_scenIdForTex`（L2763）の直後に、屋外テーマ判定のテーブルを置く
   - 初期対象は `caravan-road` のみ（既に `__daylight` 自動ON）
   - `?field=0` で強制OFF できるようにする（比較検証・撤退用）

2. **Pass 1a（L4310 天井）を屋外では空/遠景に差し替え**
   - `tile===2` のセルに `SPR_CEILING` を貼るのをやめ、代わりに:
     - **空グラデーション**: 仕様書 §3 の各フィールドの配色（例 F-02草原 `#3aa8e8` → `#6dc8f7` → `#9de0ff`）を `createLinearGradient` で
     - **遠景の丘/樹列シルエット**: `camX * 0.2` のオフセットで横方向にタイルする手続き描画（サイン波の丘＋矩形の樹列シルエット）。仕様書 §4「遠景 0.2倍速」をここで実現
   - ⚠️ 天井セルは**マップの上下と外周**にあるので、上端が「空」、下端も「空」になる点に注意。下端は地平線ではなく手前側なので、**下端は空ではなく暗い前景（草叢・土手）にする**のが自然

3. **Pass 1.2（L4435 床グリッド線）を屋外ではスキップ**
   - `rgba(0,0,0,0.22)` の 96px 軸並行格子は「見下ろしタイルゲーム」の最も明白な signature（美咲指摘）。屋外では必ず消す

4. **Pass 2（L4475 壁立て）の屋外版**
   - 屋外で「壁」を消すと `isTileWall()` が返す遮蔽と画面の見た目が食い違い、**プレイヤーが「なぜ矢が飛ばないのか」を理解できなくなる**（ガイウス指摘）
   - 消さずに、`wallPattern` を屋外テーマ用（生垣/岩肌）に差し替えるだけに留める。**STEP1 では既存 `caravan_road_wall.png` をそのまま使ってよい**

5. **`__daylight` の屋外テーマ連動**
   - 現状 caravan-road のみ自動ON（L5847）。`FIELD_MODE` と連動させる
   - ⚠️ `renderLighting` の early-return は**素の return 禁止**。必ず clear してから return

### 差分規模の見込み
index.html に **80〜130行程度**。全部 revert 可能。

### 検証
- `tools/` にヘッドレス検証ドライバ `driver_field_step1.js` を新規作成（puppeteer-core、置場は `%TEMP%/df_pptr`）
- **必須アサート**:
  1. 既存6シナリオ（goblin-mine / bandits-forest / lizard-swamp / orc-fort / undead-temple / dragon-lair）で `FIELD_MODE` が false であること
  2. 既存6シナリオの `mapCanvas.toDataURL()` ハッシュが**変更前と完全一致**すること ← **非退行の機械証明**
  3. caravan-road で `FIELD_MODE` が true、グリッド線が描かれないこと
  4. `?field=0` で従来描画に戻ること
  5. `pageerror` ゼロ、autoplay が完走すること
- スクリーンショットを **iPhone横持ち 844×390 / 縦持ち 390×844 / デスクトップ** の3種で撮り、目視判定

### changelog（必須）
`index.html` を触るので pre-commit フックが要求する。

## STEP4（実機で「奥行きが足りない」と感じた場合のみ）

案3のアフィン斜投影を、**対象を限定して**導入する。
- `DEPTH_K=1` を恒等式に落とし、既存6シナリオのスクショハッシュ一致で非退行を機械証明する設計にすること
- ⚠️ DOM側 `camY` 減算 約40箇所の置換が必要。**スクリーン空間オフセット（HPバー -10 / ラベル -27 / バッジ -46）とワールド座標を先に分類表にしてからでないと着手禁止**（内側に入れると DEPTH_K 倍に潰れる）
- ⚠️ z-index を `footWy` の生値から作らない。`z = 100 + clamp(...)` の有限バンドに正規化しないと、HUD(10) / phaseIndicator(50) / dmMessage(80) / dmNarration(100) / resultOverlay(200) を飛び越えて**キャラが DM ナレーションの上に描かれる**
- **STEP1〜3 の絵で満足なら、この STEP は永久に不要**
