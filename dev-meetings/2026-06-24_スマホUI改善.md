# 開発会議：スマホ実機のUI/レイアウト見直し（2026-06-24）

**議題**: スマホ（iOS Safari・横持ち）で ①銀の鹿亭のタイトルと更新情報パネルが重なる ②戦闘画面はテキスト領域が広すぎ・敵がフレーム外で見栄えが悪い。この2つを直す方向性を決める。
**起点**: 慶彦さんのスマホ実機スクショ2枚（戦闘HUD / 銀の鹿亭）。

## 調査で判明した動かぬ事実
- **銀の鹿亭**: `#title`（tavern.html:67）は中央寄せ・**max-width 未指定**でサブタイトルが長く右へ肥大。右上固定 `#changelogBox`（tavern.html:892, `right:20px`/`min(280px,40vw)`/`z-index:15`）と構造的に衝突。横持ちスマホ（高さ≈390px）では `@media(max-width:480px)`（731）が発火せず**モバイル最適化が一切効いていない**。
- **戦闘画面**: 左 `#partyPanel`（1863）固定280px、下 `#combatLog`（1945）固定170px。横持ち（≈844×390）で描画領域は **564×220 の小窓**（ログだけで高さの44%）。
- **二重管理**: `UI_MENU_WIDTH=280`/`UI_LOG_HEIGHT=170` が JS定数（3232-3233）とCSS（1868,1948,1950）で三重ハードコード。`updateCamera()`（3246-3283）もこの2定数依存。
- **敵が映らない**: カメラは主人公中央固定。敵は先頭から右へ1200px超。ただし交戦敵は `encounterEnemyIndices`（11038）に保持され `enemies[i].x/y` で取れる＝カメラ拡張フック有り。

# 第2段：収束会議（実装計画）

**選択案**: パネル/ログの固定chromeを縮小・ドロワー化してゲーム描画を主役にし、戦闘カメラを「味方+交戦敵」を収めるフレーミングへ拡張。酒場は横持ち対応ブレークポイントで重なりを構造的に解消。

## 第1ラウンド：設計
- **【レンツ】** 肝は `UI_MENU_WIDTH`/`UI_LOG_HEIGHT` の三重ハードコードを単一ソース化すること。CSS変数 `--ui-menu-w`/`--ui-log-h`＋JS `computeUiMetrics()` で viewport から算出しCSSへ流す。カメラの敵フレーミングは `encounterEnemyIndices` を `updateCamera` のAABBへ足すだけ。
- **【ミサキ】** 戦闘開始時に敵へズームしたい。
- **【レンツ】** 却下。DOMスプライト個別配置でズーム基盤が無く大改修。**パンのみ**。
- **【ミサキ】** わかった、ズームは将来④で。

## 第2ラウンド：肉付けと衝突
- **【ミサキ】** `partyPanel` をドロワー化し描画フル幅に。`combatLog` も170→110。
- **【ノエル】** 反対。オートバトルでHPが常に見えないのは致命的。「気づいたら全滅」になる。
- **【ミサキ】** 折衷。ドロワーは詳細用、**コンパクトHPミニバーを常設**。
- **【ノエル】** それなら賛成。タップ域48px以上。
- **【ガイウス】** ログは削っても判定ログ（DC/クリティカル）は残せ。D&Dの"らしさ"だ。トースト化はしない。
- **【ガイウス】** 酒場は横持ちで `@media(max-width:480px)` が効かないのが盲点。title に max-width、更新情報は畳んで📜化＋NEWバッジ。

## 第3ラウンド：合意
- **【レンツ】** 実装順（リスク低い順）: 1.酒場重なり解消 → 2.UI単一ソース化＋chrome縮小 → 3.カメラ再フレーム → 4.ドロワー＋HPミニバー。素材ゼロ、tavern/index間同期なし、`orientationchange`/`resize` で再計算。
- **【ノエル】** 検証は実機横/縦両方。回転追従、ドロワータップ域、ミニバーでHP一目。
