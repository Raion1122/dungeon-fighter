# 開発会議〔第1段：候補出し〕：マジックアイテム第5弾「実在の D&D マジックアイテム」

**議題**: 第5弾は独自品でなく **SRD 5.1 に実在するマジックアイテムを移植**する。どのテーマ群を最初に実装するか、方向性を1つ選ぶ。
**ゴール**: 毛色の違う候補を出して、進む方向を1つ選ぶ

**事前確認（実コードで確定した「無改修で乗る」フィールド）**:
- アクセで**完全無改修**: `skillBonus`(perception/investigation/sleightOfHand/stealth/persuasion/arcana) / `saveBonus` / `fireResist` / `freeAction` / `hpBonus` / `undeadDR` / `avoidAll`・`avoidArrow`
- **AC加算はアクセ未配線**（`currentPlayerAc`/`currentAllyAc` がアクセを読まない）→ `getSaveBonus` と同型の小配線（player/ally 2サイト）が要る
- 武器: `enhancement` / `bonusDmgType`(fire/cold/lightning/holy) / `lifeStealDice` / `critStun` / `shockwave*` / `vsUndead`
- 既に実装済で重複禁止: Flame Tongue / Frost Brand（第1弾）、Amulet of Health≒生命の護符、Ring of Free Action=自由の指輪、聖別された剣・贖罪のメイス（第2/4弾）

---

## 第1ラウンド：ブレスト（各専門領域から方向性を1つ）

【ガイウス】いいか、実在アイテム路線でまずやるべきは**王道中の王道**だ。Gold Box をやった人間なら誰でも知ってる「Ring of Protection（守りの指輪）」。AC+1 と全セーヴ+1、これぞ D&D の第一歩の宝物だ。Cloak of Protection（守りの外套）も同性能で、指輪と外套で"守りの装身具"というテーマが一枚岩になる。全職が着けられる生存アイテムってのが設定的にも綺麗だ。

【レンツ】気持ちは分かるが実装で言うと、Ring of Protection の AC+1 は**アクセの AC 加算が未配線**なんだ。`getSaveBonus` を足した時と同じ形で `getAccessoryAcBonus` を player/ally の2サイトに挿す小工事が要る。セーヴ側は完全無改修。…で、**本当に無改修で今すぐ乗るのは技能アクセ**だ。`skillBonus` は囁きの護符で実証済み。Eyes of the Eagle（鷹の目＝perception+2）と Cloak of Elvenkind（エルフのマント＝stealth+2）はデータ2ファイル足すだけで動く。工数ほぼゼロ。

【ミサキ】画の話をすると、"守り"系は発光の見せ場が作りやすいのよ。指輪や外套に淡い護りのオーラを出せば「効いてる」が伝わる。でも正直、**攻めの実在アイテム**も欲しくない？ Gauntlets of Ogre Power（大鬼の力の籠手）とか Belt of Giant Strength（巨人の力の帯）。前衛の一撃がドンと重くなる、あの分かりやすい快感。名前だけでワクワクするでしょ。

【ノエル】ちょっと待って。iPhone の片手で、電車の中で遊んでる人が「AC+1」や「全セーヴ+1」を**体感できる？** 数字が地味すぎて、着けても着けなくても分からない可能性が高い。その点、技能アクセの perception+2 は**罠発見**っていう目に見える結果に直結する。鷹の目を着けた奴が前に出て罠を見抜く——これは"効いた"が画面で分かる。攻めの籠手も、火力は第4弾の五職武器と数字が被って埋もれないか心配。

## 第2ラウンド：絞り込み（制約で間引く）

【レンツ】素材調達は全部データ内製なので外部DLゼロ、そこは全案クリア。工数で並べると **技能アクセ = 無改修（最安）＜ 守りアクセ = AC小配線（saveBonus と同型で安い）＜ 力の籠手 = ダメージ加算の小配線（player/ally 2-3サイト）**。どれも二重ファイル同期はアクセ定義の index/tavern 2箇所だけで、第2〜4弾と同じ型。

【ガイウス】破邪の名剣（Sun Blade / Mace of Disruption）はどうだ、と言いたいところだが……**第2弾の聖別された剣・贖罪のメイスと holy+vsUndead で丸被り**だな。実在度は高いが「もう持ってる剣の名前違い」になる。今回は見送り、将来アンデッド弾を拡張する時の弾にとっておこう。

【ノエル】Cloak of Elvenkind の stealth（隠密）は、正直**ゲーム内で判定が出る場面が perception ほど多くない**。鷹の目の perception（罠発見・知覚）の方が出番が確実。技能アクセを選ぶなら perception を主役、stealth は添え物にすべき。あと守りアクセを選ぶなら、AC+1 だけだと地味だから**セーヴ+1 とセット**にして「竜のブレスや畏怖に強くなる」と体感を作らないと。

【ミサキ】折衷案。技能アクセ（①）は"探索が変わる"、守りアクセ（②）は"生き残りが変わる"、力の籠手（③）は"火力が変わる"——**綺麗にジャンルが3方向に散ってる**。これなら慶彦さんが「今どれが欲しいか」で選べる。名剣は④として慎重枠で置いておけばいい。

【レンツ】異議なし。3案＋慎重枠1で決裁に回そう。

# 開発会議〔第2段：実装計画〕：豪傑の武具（能力強化の名品）

**選択案**: SRD 5.1 の STR 系名品 **Gauntlets of Ogre Power（大鬼の力の籠手）/ Belt of Giant Strength（巨人の力の帯）** を、装備中に近接ダメージが上がる gated アクセとして移植する。
**ゴール**: この案を実装に落とす開発計画書をつくる

---

## 第1ラウンド：設計

【レンツ】アーキテクチャは第2〜3弾の踏襲でいける。ダメージボーナスの権威関数は**2箇所だけ**だと確認した——リーダーが `currentPlayerDmgBonus()`(index 16588)、仲間が `effectiveAllyDmgBonus(ally)`(16574)。多段攻撃(19542)も後者を通るから自動でカバーされる。なので `getSaveBonus` と同型の `getAccessoryDmgBonus(unit)` を1つ作って、この2関数に挿すだけ。ドロップは `rollMagicItemDrop` が gated アクセを全自動収集するから**入手も無改修**。データは `accessories[]`(index 8296付近) と `ACCESSORIES_TV[]`(tavern 3859付近) の**末尾に2エントリ + id完全一致**、これが唯一の二重同期。

【ミサキ】STR 系って発動が地味になりがちだから、**殴った瞬間に力の余波**みたいなVFXを足したい。あと能力アップ系なら攻撃力の数字もドンと上げて…

【レンツ】待った。新規VFXは第4弾で「新規VFXゼロ」を貫いた路線から外れるし、iOS の描画コストも増える。既存のヒットFXで十分だ。それと**攻撃命中(to-hit)まで足すと仲間側が破綻する**——仲間の atkBonus は spawn 時に焼き込み済みで生フックが無い。ダメージは runtime 関数が2つあるから綺麗に乗るが、命中は非対称になる。**ダメージのみ**に割り切るのが工学的に正しい。

【ミサキ】うーん、じゃあ火力の"見える化"はどうするの。数字が上がっても分からなきゃ意味ないでしょ。

## 第2ラウンド：肉付けと衝突

【ノエル】そこは私も第1段で言った通り。**「効いてるのが見えない」なら受動アイテムとして失敗**。ただ新しいUI要素を足すのは回帰が怖い——ローディング画面のパネルは既存ドライバ(tavern_ui / sidebar_tier)が形を見てる。

【レンツ】それなら妙案がある。パネルの武器行は既に `formatWeaponLine(..., weaponDisplayBonus, ...)` で**武器のダメージ修正を表示している**。ここに `getAccessoryDmgBonus` を畳み込めば、「剣 1d8**+5**」の +5 に帯の +3 が乗って見える。**新UI要素ゼロ**で「自分の武器が実際に強くなった」が伝わる。しかも戦闘の実値(`currentPlayerDmgBonus`)と一致する。

【ガイウス】それでいい。D&D 的にも Gauntlets of Ogre Power は「STR を 19 にする=殴りが重くなる」だから、武器のダメージ表記に乗るのは筋が通ってる。Belt of Hill Giant Strength(STR 21)を一段上(+3)にすれば、籠手(+2)→帯(+3)の**格の違い**も出る。両方 SRD 5.1 収録、名前も一般名詞で PI 抵触なし。

【ノエル】対象職はどうする。D&D では誰でも着けられる(STR を固定値にするアイテムだから)。でもゲーム的に、**魔法使いが+3近接ダメージ**もらっても宝の持ち腐れだよね。害はないけど。ここは慶彦さんに決めてもらうところ。

【ミサキ】VFXは諦めるけど、武器行の数字が上がるのは嬉しい。妥協案として受け入れる。

## 第3ラウンド：合意

【レンツ】実装順序: ①データ2本(index+tavern末尾, id一致) → ②`getAccessoryDmgBonus` 新設 + 2サイト配線 → ③武器行表示に畳み込み(可視化) → ④ヘッドレス検証 → ⑤changelog → ⑥push。dev テストは `DF_DEV_MAGIC_SHOP` の自動付与リスト(tavern 3156)に2 id を足せば `?givemagic` 無しで所持化できる。**高水位 idx は触らない**(アクセは名前/id キーで所持管理、weaponIdx の罠とは無関係)。

【ノエル】検証は「装備で currentPlayerDmgBonus と effectiveAllyDmgBonus が +N される」「非装備は素通り」「単眼の暴君の帯(sovereign ray)上では 0 に抑制される」「武器行表示が実値と一致」を必須 assert に。

# 📋 開発計画書：豪傑の武具（能力強化の名品）

## 概要
SRD 5.1 の STR 系名品を、装備中に近接ダメージが上がる gated アクセとして2種移植する。**大鬼の力の籠手**(Gauntlets of Ogre Power → dmg +2) と **巨人の力の帯**(Belt of Hill Giant Strength → dmg +3)。第4弾までで完成した gated アクセ基盤に無改修で乗り、ダメージ配線は `getSaveBonus` と同型の小工事(2サイト)のみ。

## 実装スコープ（1フェーズで完結）
- 作る: 2種の gated アクセ(dmgBonus)、`getAccessoryDmgBonus` ヘルパー、2サイトのダメージ配線、武器行への可視化。
- 作らない: 新規VFX、to-hit ボーナス(ダメージのみ)、新スロット、AC/セーヴ効果(それは候補②の領分)。

## 流用する既存システム
- **gated アクセ基盤**: `accessories[]` / `ACCESSORIES_TV[]` / `getEquippedAccessories`(classReq・所持フィルタ込み)。
- **ドロップ**: `rollMagicItemDrop` が gated アクセを全自動収集 → `findAccessoryRecipient` が空き枠へ装着(無改修)。
- **dev テスト**: `DF_DEV_MAGIC_SHOP` 自動付与リスト(tavern 3156-3157)に2 id 追加で `?givemagic` 不要。
- **可視化**: `formatWeaponLine` の `weaponDisplayBonus` に畳み込み(新UI要素ゼロ)。
- **抑制整合**: `getAccessoryDmgBonus` 内で `sovereignSuppressed(unit)` を見て 0 を返す(他の魔法アイテムと同じく単眼の暴君のレイ上で死ぬ)。

## 新規実装
- `getAccessoryDmgBonus(unit)`: `getEquippedAccessories(unit)` を走査し `acc.dmgBonus` を合算。`sovereignSuppressed` で 0。(index 9850 `getAccessoryHpBonus` の隣)
- 配線2サイト:
  - `currentPlayerDmgBonus()`(16590): `+ getAccessoryDmgBonus("player")`
  - `effectiveAllyDmgBonus(ally)`(16574): base に `+ getAccessoryDmgBonus(ally)`(抑制時は関数が0を返すので `_dmgEnhDelta` の減算と両立)
- 可視化: 武器行の `weaponDisplayBonus` に `getAccessoryDmgBonus("player")`(リーダー)/ 仲間相当を加算表示。
- データ2本(index `accessories[]` 8296末尾 / tavern `ACCESSORIES_TV[]` 3867付近末尾, id完全一致):
  - `gauntlets-ogre-power` 「大鬼の力の籠手」 dmgBonus:2, gated, rarity:"special"
  - `belt-giant-strength` 「丘巨人の力の帯」 dmgBonus:3, gated, rarity:"special"

## 素材調達（該当時）
なし（全てデータ内製・外部DLゼロ）。

## iOS Safari 検証項目
- 新規VFX/音源ゼロ = AudioContext・描画に影響なし。
- 武器行の文字数増(+N)で折り返しが崩れないか実機確認(縦持ち)。

## 実装ステップ（順序）
1. データ2本追加(index + tavern, id一致) + dev付与リストに2 id
2. `getAccessoryDmgBonus` 新設 + 2サイト配線
3. 武器行表示へ畳み込み(可視化)
4. ヘッドレス検証(`driver_magicitem5_gauntlets.js`): 装備で+N / 非装備素通り / sovereign抑制で0 / 表示一致
5. 回帰(magicitem2/3, devmagicshop, magicweapon5, tavern_ui, sidebar_tier)
6. changelog → push

## リスクと対策
- **武器行表示が実値とズレる** → 表示は `getAccessoryDmgBonus` を戦闘と同じ関数で計算(単一ソース)。ドライバで表示==実値を assert。
- **パネル形状変化でUI系ドライバが落ちる** → 新要素は足さず既存 `weaponDisplayBonus` に数値を畳むだけ。tavern_ui/sidebar_tier を再走し、落ちたら形状 assert を更新。
- **仲間の非対称(atk焼込み)** → to-hit は実装しない(ダメージのみ)。仲間ダメージは runtime 関数 `effectiveAllyDmgBonus` に乗るので対称。
- **2アクセ同時装備で加算スタック**(D&D では STR 固定=非スタック) → 害は軽微(2枠の生成)。仕様として許容(将来必要なら最大値化)。

## 二重ファイル同期
`accessories[]`(index) ↔ `ACCESSORIES_TV[]`(tavern) の2エントリ。id・name・gated・rarity を完全一致させる。効果値(dmgBonus)は index 側だけが戦闘で読む(tavern は表示専用)。
